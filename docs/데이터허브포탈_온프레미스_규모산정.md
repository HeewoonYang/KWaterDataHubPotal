# 데이터허브 포탈 온프레미스 규모 산정

> **기준**: 한국수자원공사(K-water) 규모 공기업 (직원 8,000명)
> **환경**: 온프레미스 (물리/가상 서버)
> **범위**: 데이터허브 포탈 전용 (수집DB, 분석DB, Elasticsearch 제외)
> **작성일**: 2026-03-03

---

## 1. 포탈이 관리하는 데이터 범위

| 관리 대상 | 설명 | 예상 데이터량 |
|-----------|------|:------------:|
| 사용자/권한 | 8,000명 계정, 역할, 조직, 권한 매핑 | ~500MB |
| 메타데이터 카탈로그 | 데이터셋 목록, 컬럼 정의, 태그, 분류체계 | ~5GB |
| 데이터 활용 신청/승인 | 워크플로우, 승인이력, 신청서 | ~2GB |
| 감사/접근 로그 | 누가 언제 어떤 데이터 조회했는지 | ~50GB/년 |
| 대시보드/리포트 설정 | 사용자별 대시보드 레이아웃, 위젯 설정 | ~1GB |
| 공지/게시판/FAQ | 포탈 내 컨텐츠, 첨부파일 | ~10GB |
| 데이터 품질 지표 | 품질 점수, 이력, 프로파일링 결과 | ~5GB |
| **합계** | | **~75GB/년** |

## 2. 사용자 트래픽 산정

```
전체 직원              8,000명
  └─ 데이터허브 등록자   ~3,200명 (40%)
      └─ 일 접속(DAU)     ~800명 (25%)
          └─ 동시접속(CCU)   ~200명 (평상시)
              └─ 피크 CCU     ~500명 (전사 공지/월초 리포트)
```

| 지표 | 평상시 | 피크 |
|------|:------:|:----:|
| 동시 접속자 | 200명 | 500명 |
| TPS | 30~50 | 100~150 |
| 페이지 응답시간 목표 | ≤ 2초 | ≤ 3초 |

## 4. 서버별 규모 산정

### 4-1. WEB 서버 (Nginx/Apache)

| 항목 | 스펙 |
|------|------|
| **CPU** | 4 Core |
| **Memory** | 8GB RAM |
| **Disk** | 100GB SSD |
| **수량** | **2대** (Active-Active) |
| **OS** | Rocky Linux 8 / RHEL 8 |
| **역할** | 정적 리소스 서빙, SSL 종단, 리버스 프록시 |

> **산정근거**: Nginx 1대당 동시 연결 10,000+ 처리 가능. 500 CCU에 2대는 이중화 목적으로 충분

---

### 4-2. WAS 서버 (Tomcat/JBoss/Spring Boot)

| 항목 | 스펙 |
|------|------|
| **CPU** | 8 Core |
| **Memory** | 32GB RAM |
| **Disk** | 200GB SSD |
| **수량** | **2대** (Active-Active, 세션 클러스터링) |
| **OS** | Rocky Linux 8 / RHEL 8 |
| **JVM Heap** | 16GB 할당 |
| **Thread Pool** | 200~300 threads/인스턴스 |
| **역할** | 포탈 비즈니스 로직, 인증/인가, REST API |

> **산정근거**: WAS 1대당 Thread Pool 200~300 → 피크 500 CCU 시 2대로 분산 처리. JVM Heap 16GB 기준 32GB RAM 필요

---

### 4-3. 포탈 DB 서버 (Tibero/PostgreSQL)

| 항목 | 스펙 |
|------|------|
| **CPU** | 8 Core |
| **Memory** | 64GB RAM |
| **Disk (OS)** | 200GB NVMe |
| **Disk (Data)** | 1TB SAN (RAID-10) |
| **수량** | **2대** (Active-Standby HA) |
| **OS** | Rocky Linux 8 / RHEL 8 |
| **역할** | 사용자/권한, 메타데이터, 워크플로우, 감사로그 |

> **산정근거**: 포탈 관리 데이터 ~75GB/년 × 5년 = ~375GB → 1TB 확보. DB Buffer Cache는 Active Dataset의 2~3배 → 64GB 적정

#### 포탈 DB 주요 테이블 규모 추정

| 테이블 그룹 | 건수 (5년 누적) | 예상 용량 |
|-------------|:--------------:|:---------:|
| 사용자/권한 (user, role, org) | ~50,000건 | ~500MB |
| 메타데이터 카탈로그 | ~100,000건 | ~5GB |
| 데이터 신청/승인 | ~200,000건 | ~10GB |
| 감사/접근 로그 | ~50,000,000건 | ~250GB |
| 대시보드/위젯 설정 | ~30,000건 | ~1GB |
| 게시판/공지/FAQ | ~50,000건 | ~10GB |
| 품질 지표/이력 | ~5,000,000건 | ~25GB |
| **합계** | | **~300GB** |

### 4-4. Redis 캐시 서버

| 항목 | 스펙 |
|------|------|
| **CPU** | 4 Core |
| **Memory** | 16GB RAM |
| **Disk** | 50GB SSD |
| **수량** | **2대** (Master-Replica) |
| **역할** | 세션 관리, 권한 캐시, 메타데이터 캐시 |

> **산정근거**: 세션 8,000명 × 5KB = ~40MB, 메타데이터 캐시 ~2GB, 권한 캐시 ~500MB → 16GB로 충분

| 메타데이터 캐시 (카탈로그) | ~10,000 | ~2GB |
| API 응답 캐시 | ~5,000 | ~500MB |

## 10. 이중화 및 가용성 설계

| 구분 | 전략 | 가용성 목표 |
|------|------|:----------:|
| **WEB 서버** | Active-Active (L4) | 1대 장애 시 자동 전환 |
| **WAS 서버** | Active-Active (세션 클러스터링) | 1대 장애 시 세션 유지 |
| **DB 서버** | Active-Standby (자동 Failover) | RTO < 1분, RPO = 0 |
| **Redis** | Master-Replica (자동 전환) | RTO < 30초 |
| **스토리지** | RAID-10 + 일일 백업 | 디스크 장애 시 무중단 |
| **SLA 목표** | | **99.9% (연간 다운타임 < 8.76시간)** |

### 백업 정책

| 대상 | 주기 | 보관 기간 | 방식 |
|------|------|:---------:|------|
| DB 전체 백업 | 일 1회 (새벽 2시) | 30일 | Full Dump → NAS |
| DB 증분 백업 | 시간별 | 7일 | Archive Log |
| 첨부파일 | 일 1회 | 90일 | rsync → NAS |
| 시스템 로그 | 일 1회 | 1년 | 로그 로테이션 → NAS |

## 11. 향후 확장 시나리오

```
구축 초기              1~2년 후                3년 후
──────────────       ──────────────         ──────────────
WEB  2대        →    WEB  2대          →    WEB  2대
WAS  2대        →    WAS  3대 (증설)   →    WAS  4대
DB   2대        →    DB   2대 (스토리지↑) → DB   2대 (메모리↑)
Redis 2대       →    Redis 2대         →    Redis 3대 (센티널)
──────────────       ──────────────         ──────────────
총 8대               총 9대                  총 11대
```

| 확장 트리거 | 조건 | 조치 |
|------------|------|------|
| DAU 1,600명 초과 | WAS CPU 사용률 80%+ 지속 | WAS 1대 증설 |
| TPS 150 초과 | 응답시간 3초 초과 빈발 | WAS 증설 + DB 쿼리 튜닝 |
| DB 용량 800GB 도달 | 스토리지 80% 사용 | SAN 볼륨 확장 |
| 감사로그 급증 | 일 100만건 초과 | 로그 파티셔닝 + 아카이브 정책 |
