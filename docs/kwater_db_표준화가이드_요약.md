# K-water 데이터베이스 표준화 가이드 요약

> **원본 문서**: `한국수자원공사_데이터베이스_표준화가이드.pdf` (01-KWI-SD-SC, v1.0, 2017-04-17)
> **작성 목적**: 데이터허브포탈 ↔ 기존 데이터관리포탈 연동 시 표준 체계 참조용
> **학습일**: 2026-03-18

---

## 1. 개요

K-water 업무시스템의 데이터를 표준화하고 표준 데이터를 적용·운영하기 위한 원칙과 프로세스를 정의한 문서이다.

**대상 사용자**: 모델러(각 업무시스템 개발자), 데이터관리자(DA), DBA

---

## 2. 표준화 개요

### 2.1 데이터 표준화 정의

데이터의 한글(논리)명/영문(물리)명 명명규칙, 데이터 타입, 데이터 길이 등에 대한 기준을 마련하여 표준을 정의하는 지침 및 활동

### 2.2 데이터 표준화 목적

| 목적 | 설명 |
|------|------|
| 의사소통 명확성 및 상호운용성 증대 | 동일 데이터에 동일 명칭·정의·형식·규칙 적용, 시스템 간 상호운용성 향상 |
| 데이터 품질 향상 | 일관된 형식·규칙으로 입력 오류 최소화, 중복 정의·불일치 감소 |
| 데이터 검색 편의성 및 활용도 증대 | 신규 정보 요건 발생 시 데이터 의미·위치 신속 파악, 분석·통계 효율성 향상 |
| 데이터 공유·이행 및 관리 비용 감소 | 형식 변환·정제 비용 절감, 중복 데이터 통합, 마이그레이션 비용 감소 |
| 정보시스템 개발 생산성 향상 | 공통 기능 모듈화(Library) 재사용, 인터페이스 개발 용이 |

### 2.3 데이터 표준 관리 대상

| 구성요소 | 설명 |
|----------|------|
| **표준단어** | 최소 단위 단어. 엔티티·속성·테이블·컬럼 명칭 구성의 기본 단위 |
| **표준도메인** | 속성이 취할 수 있는 데이터 값의 범위를 용어 특성에 따라 분류한 것 |
| **표준용어** | 업무 용어를 정보시스템 기술 용어로 전환하여 일관성 있게 사용하도록 정의한 것 |

### 2.4 데이터 표준 관리 조직

| 조직 | 역할 |
|------|------|
| **모델러** | 비즈니스 요건 제시, 모델링 수행, 표준 단어·용어·도메인·코드 신규/변경 신청 |
| **데이터관리자(DA)** | 표준 체크, 모델 품질 유지, 표준 신규/변경 처리, 1차 승인, 논리/물리 통합 ERD 관리 |
| **DBA** | 물리 모델 DBMS 관련 검증, 성능 최적화, 물리 오브젝트 정합성 검사 |

## 3. 표준단어

### 3.1 표준단어 구성요소

| 구성요소 | 규칙 |
|----------|------|
| **논리명** | 최대 20자 (10자 이내 권장). 명사 사용. 복합어 가능 |
| **물리명** | 대문자 영문 알파벳. 기본 4byte (최소 2 ~ 최대 8byte). 풀네임 3byte면 그대로 사용 |
| **영문풀네임** | 물리명의 영문 풀네임 표기 |
| **분류어 여부** | 용어의 의미·데이터형식·길이를 한정하는 단어 (코드, 번호, 명, 내용, 설명, 금액, 일자 등) |
| **단어설명** | 업무 관점에서 범위와 자격 요건 명시 |

**표준단어 사전 예시:**

| 논리명 | 물리명 | 영문풀네임 | 분류어 | 설명 |
|--------|--------|------------|--------|------|
| 통계 | STATS | STATISTICS | - | 현상을 숫자로 나타냄 |
| 금액 | AMT | AMOUNT | Y | 돈의 액수 |
| 명 | NM | NAME | Y | 사람의 이름, 사물의 명칭 |

### 3.2 표준단어 정의 지침

#### 3.2.1 기본 원칙

- K-water 데이터 표준단어를 정의하고, 정보관리처 협의 후 추가 사용
- **단일어** 형태의 낱말을 표준단어로 정의하는 것이 기본 (예: 인사, 정보, 입찰)
- **명사형**으로 정의 (동사, 부사, 접속사, 조사, 복수표시, 소유격 사용 불가)
- 물리명에 **특수문자 사용 금지** (`/`, `_`, `-`, `+`, `(`, `)`)
  - 예: `E-BOOK(X)` → `EBOOK(O)`, `A/S(X)` → `AS(O)`
- 동일 의미 단어를 한글·영문 중복 정의하지 않음 (단, 범용 영문은 예외)
- 전문용어이거나 분리 시 의미 모호한 경우 해당 용어 자체를 등록 (예: 주민등록번호, 사업자등록번호)
- **축약 지양**: `사번(X)` → `사원+번호(O)`, `주민번호(X)` → `주민등록번호(O)`
- 날짜 단어는 물리 데이터 타입에 영향: `일시` → DATE형, `일자/일/월/년도/년월` → VARCHAR2형
- **두음법칙 적용**: `년도(O)`, `연도(X)`

#### 3.2.2 외래어 사용 원칙

- 가급적 한글화 원칙. 대체 한글 없거나 범용 영문은 그대로 사용 (예: SID, FTP, IP, ID)
- 외래어는 외래어 표기법에 따라 정의: `Fax(X)` → `팩스(O)`, `Email(X)` → `이메일(O)`
- 한글보다 친숙한 영문은 영문 사용, 대응 한글은 금칙어 등록: `아이피(X)` → `IP(O)`
- 영문 단어 사용 시 **반드시 대문자**: `ip(X)` → `IP(O)`
- `패스워드(X)` → `비밀번호(O)`

#### 3.2.3 동의어 사용 원칙

- 이음동의어는 대표 단어만 표준단어로 정의, 나머지는 **금칙어**로 등록
  - `이름(X)` → `명(O)`, `스케쥴(X)` → `스케줄(O)`

#### 3.2.4 분류단어 원칙

- 용어의 의미와 데이터 형식·길이를 한정하는 단어 (도메인의 근간)
- 속성값 유형을 명확하게 하기 위해 속성명 뒤에 붙이는 단어
- 예: **코드, 번호, 명, 내용, 설명, 금액, 일시**

## 4. 표준도메인

### 4.1 개요

- 속성이 취할 수 있는 값의 집합
- 용어의 마지막 단어(분류단어)를 기준으로 도메인화
- 예: `~ID`, `~명`, `~금액`, `~내용`, `~일시`

### 4.2 구성요소

| 구성요소 | 설명 |
|----------|------|
| **도메인그룹** | 성격 유사한 도메인들의 관리 단위 |
| **도메인명** | `[수식어 + 구분(N/V/C/D/CL/BL) + 길이]` 형태 |
| **도메인유형** | 범위형(숫자), 열거형(텍스트), 형식형(코드) |
| **논리 데이터 타입** | NUMBER, VARCHAR2, CHAR, DATE, CLOB, BLOB |
| **데이터 길이** | 도메인 값의 최대 길이 |
| **소수점** | 소수점 이하 최대 자릿수 |

### 4.3 도메인그룹 분류

| 도메인그룹 | 정의 | 예시 |
|------------|------|------|
| 금액 | 화폐·상품 가치 표현 | 가격, 금액, 비용, 세금, 요금 |
| 비율 | 비교값 표현 | 율, 비율, 환율, 연체율 |
| 수량 | 개수·량 표현 | 수, 량, 점수, 수위, 연령 |
| 차례 | 값의 순서 표현 | 순위, 레벨, 순번 |
| 명 | 문자 형식 식별 | 명 |
| 텍스트 | 자유 형식 텍스트 | 내용, 요약, 비고, 설명 |
| 번호 | 특정 의미 체계 관리 | 사업자등록번호, ID, 식별자, 번호, 전화번호 |
| 날짜 | 시점·시간 표현 | 일자, 일, 월, 년도, 년월 |
| 주소 | 연락·접속 주소 | 우편번호, 전화번호, URL, IP, 주소 |
| 코드 | 분류·구분 속성 | 공통코드, 표준분류코드 |
| 분류 | 여부(Y/N), 유무(1/0) | 결혼여부, 사용여부, 가족유무 |
| 기타 | 유형 모호한 데이터 | 값, 암호화, 하한값, 상한값 |

### 4.4 표준도메인 사전 예시

| 도메인그룹 | 도메인명 | 유형 | 논리 타입 | 길이 | 소수점 |
|------------|----------|------|-----------|------|--------|
| 금액 | 금액N13 | 범위형 | 숫자 | 13 | - |
| 수 | 수N | 범위형 | 숫자 | - | - |
| 명 | 명V100 | 열거형 | 가변길이문자 | 100 | - |
| 수량 | 평가점수N6,3 | 범위형 | 숫자 | 6 | 3 |
| 분류 | 여부C1 | 형식형 | 고정길이문자 | 1 | - |

## 5. 표준용어

### 5.1 정의

- 논리 데이터 모델링 시 속성(Attribute)명에 사용
- 물리 전환 시 영문 컬럼명 생성의 기준

### 5.2 명명규칙

#### 5.2.1 한글명(Attribute) 규칙

- **1개 이상의 표준단어 + 분류단어**로 구성
  - `주소(X)` → `사용자주소(O)`, `내용(X)` → `게시물내용(O)`
- 한글 글자수는 영문 전환(30자리)을 고려하여 표현
- **띄어쓰기 및 특수문자 사용 금지**
- 암호화 적용 컬럼: `논리명###암호화` / `물리명###_ENC`
  - 예: `사용자주민등록번호암호화` → `USR_RRN_ENC`
- 주체 불분명 시 주제 포함: `주민등록번호` → `사용자+주민등록번호`

#### 5.2.2 영문명(Column) 규칙

- 한글명을 단어사전 참조하여 영문명으로 변환
- 영문 단어 사이 **언더바(`_`)** 사용: `USR_ID(O)`, `USRID(X)`
- 하나의 용어는 동일한 영문명·데이터타입·길이를 가짐
  - `사용자ID` → `USR_ID(O)`, `USER_ID(X)`
- 숫자 결합 예시:
  - `내용(CTNT)` → `게시물내용` → `NTT_CTNT1`, `NTT_CTNT2`
  - `주소(ADDR)` → `사용자주소` → `USR_ADDR1`, `USR_ADDR2`
- **분류도메인(여부/유무) 사용 용어**: 항상 `NOT NULL` 적용

## 6. 데이터 오브젝트 명명규칙

### 6.1 논리적 데이터 명명규칙

#### 6.1.1 엔티티 명명규칙

- **업무구분코드 + `_` + 엔티티명** 형식
  - 예: `SC_공통코드`, `SC_사용자정보`
- 업무구분코드는 단위시스템별 구분코드 참조

#### 6.1.2 속성 명명규칙

- 5.2.1 용어의 한글명 규칙 참조

### 6.2 물리적 데이터 명명규칙

#### 6.2.1 테이블 명명규칙

**구조**: `[물리특성구분어][논리특성구분어]_[업무구분코드]_[영문테이블명]`

**물리적 특성 구분어:**

| 구분어 | 의미 |
|--------|------|
| T | 테이블 |
| V | 뷰 |
| P | 파티션 |

**논리적 특성 구분어:**

| 구분어 | 유형 | 설명 |
|--------|------|------|
| M | 마스터/기본/일반 | 현재·최종 상태의 기본 정보, 주제영역 대표 엔티티 |
| D | 상세/내역 | 기본 엔티티에 대해 1:M으로 상세 정보 관리 |
| H | 이력/로그 | 과거 정보를 시점별로 보관 |
| S | 집계/통계 | 통계·합계 산출용 |
| T | 임시 | 임시 사용 |
| C | 코드 | 코드성 테이블 |
| I | 연계 | 대외기관/SAP 연계용 인터페이스 |
| B | 백업 | 과거 데이터 보존용 |
| E | 기타 | 위 유형 외 |

**예시:**
- 공통 주제영역: `TM_SC_USR_INFO` (사용자정보)
- 일반 주제영역: `TM_CS10002` (`T` + `M` + `_` + 업무코드 + 일련번호)
- 테이블 명칭은 `_` 포함 **25byte 이내**

#### 6.2.2 컬럼 명명규칙

- 5.2.2 용어의 영문명(Column) 규칙 참조

#### 6.2.3 기타 물리 데이터 명명규칙

| 오브젝트 | 명명 패턴 | 예시 |
|----------|-----------|------|
| **DB명** | `[시스템명]DB` | `KWATERDB` |
| **Tablespace** | `TS_[시스템코드]_[D/I/LOB][번호]` | `TS_ADT_D01`, `TS_ADT_I01` |
| **Data File** | `[Tablespace명]_[번호].DBF` | `TS_ADT_D01_001.DBF` |
| **파티션** | `P[논리구분]_[테이블명]_[번호]` | `PM_SC_USR_INFO_01` |
| **인덱스** | `[테이블명]_IX[번호]` 또는 `_PK` | `TN_SC_USR_INFO_IX01`, `TN_SC_USR_INFO_PK` |
| **뷰** | `V[논리구분]_[테이블명]_[번호]` | `VM_SC_USR_INFO` |
| **시퀀스** | `SQ_[테이블명]_[컬럼명]` | `SQ_SC_USR_INFO_USR_ID` |
| **Role** | `RL_[Object관리자]_[CRUD]` | `RL_SCM_CRUD` |
| **Constraint** | `[테이블명]_[참조테이블]_FK[번호]` | `COMM_USR_ATHR_FK01` |
| **Sub Program** | `FN/SP/PC/PB_[업무코드]_[명칭]` | `FN_SC_USR` |
| **User명** | `[시스템코드]` | `SCM`, `ADT`, `HRM`, `FMS` 등 |

**기타 규칙:**
- TABLE, COLUMN의 COMMENT는 항상 **논리명**으로 생성
- TABLE명과 동일한 **PUBLIC SYNONYM** 생성
- 테이블 영역과 인덱스 영역의 Tablespace는 **항상 분리**

## 8. 대용량 데이터 관리

### 파티션 테이블 기준

- 데이터 Size **10GB 이상**이고 **월 30만건 이상** 발생하는 테이블
- OLTP 소량 데이터 조회 시에는 지양

| 파티션 종류 | 설명 |
|-------------|------|
| 범위(Range) | 범위 조건 조회 시 유용, 관리 용이 |
| 해시(Hash) | 범위로 파티션할 수 없는 데이터에 적합, 성능 유리 |
| 리스트(List) | 컬럼 값이 이산 집합인 데이터에 적합 |
| 복합(Composite) | 큰 파티션의 I/O 요청을 분산 (Main-Sub) |

### BLOB 데이터 관리

- 대용량 BLOB 데이터는 **File로 변환**하여 WAS단 FileSystem에서 관리
- DB Table의 첨부파일/첨부파일상세에서 **메타정보만 관리**

## 10. 데이터허브포탈 연동 시 주요 고려사항

### 10.1 명명규칙 매핑

데이터허브포탈(`portal_schema.sql`)은 자체 표준약어 체계(`db/std_new_words.md`)를 사용하므로, 기존 데이터관리포탈의 표준단어와 **매핑 테이블**이 필요하다.

| 항목 | 기존 데이터관리포탈 | 데이터허브포탈 |
|------|---------------------|---------------|
| 테이블 접두어 | `TM_`, `TD_`, `TH_` 등 (물리+논리 구분어) | 없음 (snake_case 소문자) |
| 컬럼 물리명 | 대문자 (예: `USR_ID`) | 소문자 (예: `usr_id`) |
| 업무구분코드 | 테이블명에 포함 (예: `TM_SC_USR_INFO`) | 포함하지 않음 (예: `usr_acnt`) |
| 약어 체계 | K-water 표준사전 기반 | `std_new_words.md` 기반 |
| 감사 컬럼 | 별도 정의 필요 | `crtd_at`, `crtd_by`, `updtd_at`, `updtd_by` 공통 |

### 10.2 데이터 타입 호환

- 기존 포탈: Oracle 기반 (NUMBER, VARCHAR2, CHAR, DATE, CLOB, BLOB)
- 데이터허브포탈: PostgreSQL 기반 (INTEGER, VARCHAR, TEXT, TIMESTAMP, JSONB 등)
- 연동 시 **타입 변환 매핑** 필요

### 10.3 도메인 체계 연동

- 기존 포탈의 도메인그룹 (금액, 비율, 수량, 날짜 등)과 데이터허브포탈의 ENUM 타입 간 매핑
- 특히 코드성 데이터는 `code_group`/`code` 테이블과의 정합성 확인 필요

### 10.4 개인정보 보호

- 기존 포탈의 암호화 정책(SEED/SHA256)과 데이터허브포탈의 비식별화 정책(`deidentify_rule_type` ENUM) 간 정합성 확인
- 연동 데이터에 개인정보가 포함될 경우 양측 암호화 정책 준수 필요
