# 데이터허브 포탈 클라우드 네이티브 규모 산정

> **기준**: 한국수자원공사(K-water) 규모 공기업 (직원 8,000명)
> **환경**: 클라우드 네이티브 (Kubernetes 기반 컨테이너)
> **범위**: 데이터허브 포탈 전용 (수집DB, 분석DB 제외)
> **작성일**: 2026-03-03

---

## 1. 포탈이 관리하는 데이터 범위

| 관리 대상 | 설명 | 예상 데이터량 |
|-----------|------|:------------:|
| 사용자/권한 | 8,000명 계정, 역할, 조직, 권한 매핑 | ~500MB |
| 메타데이터 카탈로그 | 데이터셋 목록, 컬럼 정의, 태그, 분류체계 | ~5GB |
| 데이터 활용 신청/승인 | 워크플로우, 승인이력, 신청서 | ~2GB |
| 감사/접근 로그 | 누가 언제 어떤 데이터 조회했는지 | ~50GB/년 |
| 대시보드/리포트 설정 | 사용자별 대시보드 레이아웃, 위젯 설정 | ~1GB |
| 공지/게시판/FAQ | 포탈 내 컨텐츠, 첨부파일 | ~10GB |
| 데이터 품질 지표 | 품질 점수, 이력, 프로파일링 결과 | ~5GB |
| **합계** | | **~75GB/년** |

## 2. 사용자 트래픽 산정

```
전체 직원              8,000명
  └─ 데이터허브 등록자   ~3,200명 (40%)
      └─ 일 접속(DAU)     ~800명 (25%)
          └─ 동시접속(CCU)   ~200명 (평상시)
              └─ 피크 CCU     ~500명 (전사 공지/월초 리포트)
```

| 지표 | 평상시 | 피크 |
|------|:------:|:----:|
| 동시 접속자 | 200명 | 500명 |
| TPS | 30~50 | 100~150 |
| 페이지 응답시간 목표 | ≤ 2초 | ≤ 3초 |

## 4. 컨테이너(Pod) 규모 산정

### 4-1. 포탈 Frontend (Nginx + SPA)

| 항목 | 스펙 |
|------|------|
| **컨테이너 리소스** | 0.5 vCPU / 512MB RAM (request) |
| **Limit** | 1 vCPU / 1GB RAM |
| **Pod 수 (평상시)** | 2개 |
| **Pod 수 (피크)** | 4개 (HPA 자동 스케일링) |
| **역할** | 정적 리소스 서빙, SPA 호스팅 |
| **HPA 기준** | CPU 70% 또는 RPS 500 초과 시 스케일아웃 |

### 4-2. 포탈 Backend API (Spring Boot / Node.js)

| 항목 | 스펙 |
|------|------|
| **컨테이너 리소스** | 2 vCPU / 4GB RAM (request) |
| **Limit** | 4 vCPU / 8GB RAM |
| **Pod 수 (평상시)** | 2개 |
| **Pod 수 (피크)** | 4개 (HPA 자동 스케일링) |
| **역할** | 인증/인가, 비즈니스 로직, REST API |
| **HPA 기준** | CPU 70% 또는 TPS 50/Pod 초과 시 스케일아웃 |

> **산정근거**: Pod 1개당 Thread 100~150 처리 → 피크 500 CCU 시 Pod 4개로 분산

### 4-3. 백그라운드 Worker (비동기 처리)

| 항목 | 스펙 |
|------|------|
| **컨테이너 리소스** | 1 vCPU / 2GB RAM (request) |
| **Limit** | 2 vCPU / 4GB RAM |
| **Pod 수** | 1~2개 |
| **역할** | 승인 알림, 감사로그 적재, 스케줄 배치 |

## 5. Managed Service 규모 산정

### 5-1. 포탈 DB (Managed RDS / Tibero Cloud)

| 항목 | 스펙 |
|------|------|
| **인스턴스 타입** | 4 vCPU / 32GB RAM |
| **스토리지** | 500GB SSD (자동 확장 활성화) |
| **구성** | Primary + Standby (Multi-AZ HA) |
| **IOPS** | 3,000 IOPS (Provisioned) |
| **백업** | 자동 일일 백업 + 7일 보관 |
| **역할** | 사용자/권한, 메타데이터, 워크플로우, 감사로그 |

> **산정근거**: 포탈 데이터 ~75GB/년 × 5년 = ~375GB → 500GB로 여유 확보, Multi-AZ로 가용성 99.95%

### 5-2. Redis (Managed Cache)

| 항목 | 스펙 |
|------|------|
| **인스턴스 타입** | 2 vCPU / 8GB RAM |
| **구성** | Primary + Replica (자동 Failover) |
| **역할** | 세션 관리, 권한 캐시, 메타데이터 캐시 |

> **산정근거**: 세션 8,000명 × 5KB = ~40MB, 캐시 ~2GB → 8GB 충분

### 5-3. Elasticsearch (Managed Search)

| 항목 | 스펙 |
|------|------|
| **인스턴스 타입** | 2 vCPU / 16GB RAM |
| **스토리지** | 200GB SSD |
| **노드 수** | 2노드 (검색 가용성 확보) |
| **역할** | 데이터 카탈로그 전문검색, 로그 검색 |

> **산정근거**: 메타데이터 인덱스 ~10GB, 검색 QPS < 50 → 2노드 충분

## 7. Kubernetes 클러스터 노드 산정

### 전체 리소스 합산

| 구성 요소 | vCPU (request) | RAM (request) | Pod 수 |
|-----------|:--------------:|:-------------:|:------:|
| Frontend Pod | 0.5 × 2~4 = 1~2 | 512MB × 2~4 = 1~2GB | 2~4 |
| Backend Pod | 2 × 2~4 = 4~8 | 4GB × 2~4 = 8~16GB | 2~4 |
| Worker Pod | 1 × 1~2 = 1~2 | 2GB × 1~2 = 2~4GB | 1~2 |
| 시스템 Pod (모니터링 등) | ~2 | ~4GB | 3~5 |
| **합계 (평상시)** | **~8 vCPU** | **~15GB** | **~8** |
| **합계 (피크)** | **~14 vCPU** | **~26GB** | **~13** |

### Worker Node 구성

| 항목 | 스펙 |
|------|------|
| **노드 스펙** | 8 vCPU / 32GB RAM (per node) |
| **노드 수 (최소)** | **3대** (가용영역별 1대, HA 보장) |
| **노드 수 (권장)** | **4대** (여유 확보 + 롤링 업데이트 대응) |
| **Auto Scaling** | 최소 3대 → 최대 6대 (Cluster Autoscaler) |

> **산정근거**: 피크 14 vCPU / 26GB → 노드 3대(24 vCPU / 96GB)면 활용률 ~60%. 노드 1대 장애 시에도 2대로 서비스 유지 가능

## 10. Auto Scaling 정책

### HPA (Horizontal Pod Autoscaler)

```yaml
# 포탈 Backend HPA 예시
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: portal-backend-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: portal-backend
  minReplicas: 2
  maxReplicas: 4
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "50"
```

### Cluster Autoscaler

| 설정 | 값 |
|------|------|
| 최소 노드 | 3대 |
| 최대 노드 | 6대 |
| 스케일아웃 조건 | Pending Pod 발생 시 |
| 스케일인 조건 | 노드 활용률 50% 미만 10분 지속 |

## 11. 가용성/재해복구 설계

| 구분 | 전략 | 목표 |
|------|------|------|
| **애플리케이션** | Pod 다중 복제 + PDB | 무중단 배포 |
| **데이터베이스** | Multi-AZ Standby + 자동 Failover | RTO < 1분, RPO = 0 |
| **캐시 (Redis)** | Primary-Replica 자동 전환 | RTO < 30초 |
| **백업** | DB 일일 자동 백업 7일 보관 | RPO ≤ 24시간 (풀백업) |
| **DR** | 타 리전 DB 복제 (선택) | RTO < 1시간 |
| **SLA 목표** | | **99.9% (연간 다운타임 < 8.76시간)** |

## 12. 향후 확장 시나리오

```
구축 초기              1~2년 후                  3년 후
──────────────       ──────────────           ──────────────
K8s Node 3~4대  →    K8s Node 4~5대       →   K8s Node 5~6대
Backend Pod 2개 →    Backend Pod 3개      →   Backend Pod 4개
DB 4vCPU/32GB  →    DB 8vCPU/64GB (스케일업) → DB 8vCPU/64GB
Redis 8GB      →    Redis 16GB            →   Redis 16GB
ES 2노드       →    ES 3노드              →   ES 3노드
──────────────       ──────────────           ──────────────
월 ~1,000만원        월 ~1,400만원             월 ~1,700만원
```

> **확장 트리거**: DAU 1,600명 초과 또는 평상시 TPS 100 초과 시 Backend Pod 증설 우선
